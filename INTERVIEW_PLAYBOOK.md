# 提问脚手架 · Interview Playbook

本文件是面试官模式（模式 A）的"弹药库"。Agent 在追问时按**层级 → 领域**选问题。每个问题后面括号里是"出题意图"，用于判断用户答得合不合格。

---

## L1 场景识别 + 选型 Trade-off

### 1.1 通用问题（任何项目都适用）

1. **这个项目解决的业务问题用一句话说是什么？用户/客户是谁？**（定义能力）
2. **预期规模是多少？**（DAU / QPS / 数据量 / 并发）→ 如果用户答不上来，这是 **⚠️ 红旗**
3. **启动项目时有哪些约束？**（时间、人力、预算、已有技术栈、合规）
4. **你在项目中的角色和决策边界是什么？**（区分"我做了" vs "团队做了"）
5. **同样的需求，你考虑过哪几种方案？最后选了哪个？为什么？**（核心！trade-off）
6. **如果今天重做，你会改哪里？**（Learning，PARL 的 L）

### 1.2 选型拷问模板（for 每个具体技术决策）

对每一个选型（框架/库/架构模式/数据库/协议…），按下列顺序追问：

```
a) 你选了 [X]。候选还有 [Y] 和 [Z]，为什么不选它们？
b) 你选 [X] 是基于哪些指标/场景？
c) [X] 的局限是什么？什么情况下你会后悔这个选择？
d) 未来如果规模涨 10 倍，[X] 还撑得住吗？拐点在哪？
```

**示例**：
- 选了 Redis 做缓存 → "为什么不用 Memcached？为什么不用本地进程内缓存？Redis 集群规模撑到多大 QPS 就得分片？"
- 选了 React → "为什么不 Vue？项目里 React 的哪些特性被你用到了（如果只用到最基础功能，选型不成立）？"
- 选了 MongoDB → "你这个数据是强关联的还是弱关联？为什么不用 PostgreSQL？事务一致性需求怎么满足？"

### 1.3 模糊表达拦截器（遇到就拦）

用户说出下列词，立刻追问括号里的问题：

| 模糊词 | 立刻追问 |
|---|---|
| "大概""应该""可能" | "怎么测的？数据从哪来？"|
| "性能好" | "哪个指标？P99 还是均值？基准是什么？" |
| "高并发" | "多少 QPS？峰值/均值？写多还是读多？" |
| "稳定" | "SLA 几个 9？有过事故吗？MTTR 多久？" |
| "优化了" | "优化前后的数字分别是多少？怎么测的？" |
| "用户体验好" | "哪个指标？LCP/FID/CLS？怎么量化的？" |
| "安全" | "防什么攻击？威胁模型是啥？" |
| "易扩展" | "下一版要加 XX 功能，你的架构怎么容纳？" |

## L2 实现细节

### 2.1 通用问题

1. **画一下数据流/请求链路**（从用户点击到响应返回，每一跳做了什么）
2. **核心数据结构/表结构是什么？为什么这么设计？**
3. **请求失败的时候走哪条路径？**（降级/重试/熔断/回滚）
4. **并发/并行边界在哪？**（锁、幂等、竞态）
5. **你这个模块的测试策略？**（单测/集成/压测/混沌）
6. **哪块代码你自己都觉得写得不好？**（自我认知信号）

### 2.2 按领域的深度问题

**前端**
- 渲染：SSR/CSR/SSG 的选型理由？水合（Hydration）的时机和坑？
- 状态管理：为什么选 X（Redux/Zustand/MobX）？全局 state 的边界是什么？
- 路由：动态路由怎么处理 SEO？
- 构建：Bundle 体积怎么控制？代码分割策略？
- 弱网/断网场景怎么处理？

**后端**
- 接口设计：RESTful 还是 RPC？为什么？幂等性怎么保证？
- 事务：分布式事务怎么处理？2PC/TCC/Saga/本地消息表 的选型？
- 缓存：缓存穿透/击穿/雪崩怎么防？缓存一致性方案？
- 限流：令牌桶/漏桶/滑动窗口？放哪一层？
- 鉴权：JWT/Session 选型？刷新策略？

**数据库**
- 索引设计的依据？最左匹配用到了吗？有没有覆盖索引？
- N+1 怎么避免的？慢查询 Top 10 是哪些？
- 主从延迟多少？读写分离怎么处理一致性？
- 数据量涨到 1 亿怎么办？分库分表的 key 怎么选？

**AI / Agent（如本项目）**
- Prompt 工程：System Prompt 多长？变量怎么管理？版本怎么迭代？
- Token 成本：单次请求均值 tokens？月成本？有没有做缓存降本？
- 防幻觉：工具调用参数怎么校验？多轮限制怎么设？
- 流式：SSE/WebSocket 选型？断连恢复策略？

**分布式 / 部署**
- 扩容策略：水平还是垂直？无状态化改造做了哪些？
- 监控：哪些指标进 Dashboard？告警阈值怎么定？
- 灰度：Canary/A-B 怎么做的？回滚按钮按下多久能生效？

**安全**
- 威胁模型：你假设攻击者能做什么？不能做什么？
- OWASP Top 10 里涉及哪几项？怎么防的？
- 数据脱敏：什么数据需要脱敏？脱敏在哪一层做？

## L3 指标与测量（重中之重）

### 3.1 量化指标追问四连

**对用户说出的每一个指标，按下面四个问题追**：

```
a) 这个指标具体是什么？分母分子是什么？
b) 用什么工具测的？工具的测量原理是什么？
c) 你自己能手动复现这个测量吗？
d) 这个指标有 bad case 吗？什么时候测不准？测不准怎么办？
```

**⚠️ 第 (c) 点是分水岭**。初级候选人只会说"用 Lighthouse 测的"，高级候选人能讲到"LCP 本质是 PaintTimingAPI 在 layout 完成后取最大的图片/文本元素的绘制时间戳，用 PerformanceObserver 监听 `largest-contentful-paint` 事件就能自己实现"。

### 3.2 常见指标的"原理 + bad case"对照

| 指标 | 原理摘要 | 典型 bad case |
|---|---|---|
| **LCP** | PerformanceObserver 监听 `largest-contentful-paint`，取渲染稳定前最大的可见元素绘制时间 | ① 用户切 tab，timing 冻结；② LCP 元素动态变化；③ `content-visibility: hidden` 元素不算 LCP；④ 跨域图片 timing 被 mask |
| **FID / INP** | 首次输入延迟 / 全量交互延迟。INP 更准，取交互从开始到下一帧绘制的时间 P98 | ① 没有交互的页面直接 0；② passive listener 时间不算 |
| **CLS** | 布局偏移的加权累加（影响面积 × 距离），session 窗口 5s | ① 用户主动交互后 500ms 内的偏移不计；② iframe 内偏移不计入主文档 |
| **TTFB** | 从 navigationStart 到首字节到达 | CDN 命中 vs 回源差异巨大，要分桶 |
| **P99 延迟** | 99 分位耗时 | ① 样本量不够时抖动剧烈；② 需要排除冷启动/预热请求 |
| **QPS** | 每秒请求数 | ① 均值掩盖 burst；② 要分 endpoint 看，单接口 QPS 不代表系统容量 |
| **缓存命中率** | hit / (hit + miss) | ① 热 key 撑起全局命中率，但长尾查询全 miss；② 要看 key 分布 |
| **GC Pause** | JVM/Node 垃圾回收 stop-the-world 时间 | ① Young GC 频率比时长更重要；② 容器 CPU throttle 会放大 pause |
| **SSE 流式延迟** | 从 server flush 到 client 收到首个 chunk | ① 中间代理（Nginx、CDN）缓冲；② 浏览器对 text/event-stream 的解析时机 |
| **AI Token/Cost** | 单次请求总 tokens × 单价 | ① 隐藏的 system prompt 也算；② function calling 参数也算；③ 缓存命中的 input tokens 定价不同 |

### 3.3 "手动复现测量"的分级题

按用户想面的级别，问到对应深度：

- **初级**：能打开 Chrome DevTools Performance 面板看 LCP 数字
- **中级**：能写 PerformanceObserver 代码自己上报
- **高级**：能解释浏览器内核什么时候触发这个事件，能识别 DevTools 和真实用户数据（RUM）差异
- **Staff**：能指出 spec 里的模糊地带（如 INP 的 soft navigation 处理），并说明你们怎么应对

## L4 外延 / 八股文

只在以下情况触发 L4：
- 用户主动问"还有什么可能被问到的"
- 亮点 L1-L3 跑完后用户还有精力
- 时间预算还剩 >20%

**不要主动深入 L4**，原则是"点名 + 指路"：指出涉及的八股主题，链接到 [QUESTION_BANK.md](QUESTION_BANK.md) 里对应小节，让用户自己背。

示例回复：
> "你这个 SSE 方案还会被追问到：① TCP/HTTP 长连接的区别；② HTTP/2 Server Push vs SSE；③ 反向代理缓冲对流式的影响。这三个属于八股文，我不细问了，你看 QUESTION_BANK 里的`流式与长连接`小节自己过一遍。"

## 典型陷阱（Poke Holes）

面试官最爱设的 8 个陷阱。Agent 在追问时**主动埋**这些问题：

### 陷阱 1：「这是我们团队做的」
→ 追问："其中具体哪几行代码 / 哪个模块是**你个人**推动/实现的？"

### 陷阱 2：「数据是我估算的」
→ 追问："估算依据？没有真实压测你怎么敢上线？"

### 陷阱 3：「我们选了行业最佳实践」
→ 追问："'最佳实践'是什么语境下的？你们的场景有没有可能不适用？举个反例。"

### 陷阱 4：「性能提升 80%」（无基准）
→ 追问："从多少提升到多少？基准机器配置？并发数？"

### 陷阱 5：「我们用了 XX 开源库」（未读源码）
→ 追问："你看过它核心模块的源码吗？讲一下它的 [核心机制]（如 React fiber / Redis 事件循环 / Nginx worker 模型）"

### 陷阱 6：「上线后没出过问题」
→ 追问："真的吗？监控覆盖了哪些指标？你怎么知道没出过？举个你处理过的 bug。"

### 陷阱 7：「这个我们还没做」
→ 追问（友好版）："如果今天让你做，你的方案是什么？三种实现方式分别是什么？"（评估规划能力）

### 陷阱 8：「这块是前端/后端的事」
→ 追问："跨端的接口你怎么跟对方对齐的？对方给你挖过坑吗？"（协作维度）

## 提问节奏模板（Agent 可直接套用）

### 节奏 A：开场（第 1 轮问答）
```
好，我们来看亮点 <名字>。先打基础：
1. 用一句话说这个解决的业务问题是什么？
2. 项目上线时预期规模是多少？（DAU / QPS / 数据量选一个）
3. 这块是你从 0 到 1 做的，还是在现有基础上改的？你的贡献边界？

先答这三个，答完我看情况追问。
```

### 节奏 B：选型深挖（第 2 轮）
```
OK，刚才说你选了 <X>。我想追两层：
1. 候选方案还有哪些？按哪些维度 PK 的？（trade-off 表格）
2. 选 X 你当时最担心的风险是什么？实际碰到过吗？

如果两个都能答出来，我们进实现层。
```

### 节奏 C：指标拷问（第 3-4 轮）
```
上面你提到 <指标 N = 数字>。四连发：
1. 这个数字的分子分母是什么？
2. 什么工具测的？
3. 不用工具，你能从原理层面自己测一次吗？
4. 这个指标有测不准的场景吗？

这是面试高频追问，务必答到第 3 点。
```

### 节奏 D：总结追问（某亮点收尾）
```
这个亮点差不多了。最后三个问题收尾：
1. 如果今天重做，你会怎么改？（Learning）
2. 规模涨 10 倍，你现在的方案哪里会先崩？
3. 你这个方案相比开源标杆（<列出对标>）的差距是什么？

答完我们切下一个亮点 / 或者我给你生成这个点的 STAR 文档。
```

## 回答降级三档（用户答不上来时的处理）

这是 SKILL.md "用户挫败感兜底铁律"的**话术落地**。当用户回答出现下面关键词时，按对应档位响应。

### 档位 1：含糊表达（可救）
**触发词**："大概""应该""差不多""好像是""印象中"

**处理**：不降级，正常追问（用"模糊表达拦截器"那一节的话术）。不要给答案，逼用户自己想清楚。

### 档位 2：半懂（给提示）
**触发词**："不太确定""这个我没深想""方向是 X 但细节记不清"

**处理**：给**指路提示**而不是答案。
```
话术模板：
"这个点的关键在 <角度 A> 和 <角度 B>，你从哪个角度切入都行。
你看一下项目里 <具体文件路径>，那个 <关键词> 是个 hint。
再想 2 分钟，想不出来我直接讲。"
```

### 档位 3：完全不会（直接讲）⭐ 兜底触发
**触发词**："不知道""没想过""不清楚""没这方面想法""跳过吧"

**处理**：**立即停止追问**，切讲解模式。标准话术：

```
"OK 这个点我直接讲——[不说"这很基础"或"你应该知道"]

【结论一句话】：<核心答案>

【要点】：
1. <要点 1，带原理>
2. <要点 2，带 trade-off>
3. <要点 3，带 bad case 或数据>

【为什么面试会问】：<1 句说明考察意图>

现在你用自己的话复述一遍核心逻辑，不用完全对，关键是抓住 <1 个核心点>。"
```

**讲解完必做的 3 件事**：
1. 让用户复述核心（验证真的懂了）
2. 让用户在**项目里找一行相关代码**（把理论落到项目上，避免变成纯背书）
3. 内部记录：`⚠️ 需复习：<具体点>`，最终文档里标注

### 档位 4：连续触发（进模式 C）

**同一亮点内连续 2 次进入档位 3** → 立刻进入 SKILL.md 的模式 C（教练讲解模式）：

```
"打断一下——我们换个节奏。这个亮点涉及的基础概念比较多，
我先花 5 分钟给你系统讲一遍背景和架构，你跟着我梳理完再回来答问题，
效果会比硬答好很多。可以吗？"
```

用户同意后，按 [EXAMPLES.md](EXAMPLES.md) 的"模式 C 讲解节奏"走（背景 → 主流方案对比 → 架构 → 指标铺垫 → 复述检验）。

## 搜索辅助话术

SKILL.md 的"主动网络搜索"章节告诉 Agent **何时搜**，这里是**搜完怎么用**的具体话术。

### 话术 A：在追问里嵌入搜索结果

```
"你选了 X。我刚查了一下，2026 年业界常见对标是 Y 和 Z，
Y 的卖点是 <特性 A>，Z 的卖点是 <特性 B>。
你的 X 方案和这两个比，优势和劣势各是什么？
如果你没对比过，就从 <1 个维度> 切入说也行。"
```

### 话术 B：针对目标公司的追问

（当用户提供了目标公司/部门，搜索其技术栈后）

```
"你面 <公司> 的 <部门>。我查到他们最近在用 <技术 A> 和强调 <方向 B>。
你项目里对应的是 <用户项目里的点>，差异是 <差异点>。
如果面试官问你'为什么不直接用 <技术 A>'，你怎么答？"
```

### 话术 C：引用权威信源补强指标

```
"你说 LCP 优化到 280ms。据 Google web.dev 2026 年的基准，
<2.5s 算 Good，<1s 算 Excellent。你这个数字落在 Excellent 区间，
但面试官可能会质疑测量环境。你当时是 4G 模拟还是 WiFi？
单次测量还是 P75？"
```

### 搜索引用的标注格式

文档产出时，如果某个论断来自搜索，在括号里标注：
- `（据 Google web.dev 2026）`
- `（据 GitHub 2026 年 JavaScript 框架趋势报告）`
- `（综合 1point3acres 2026 字节前端面经 10 篇）`

**不要**编造来源或写"据调研"这种虚词。

## Agent 自检清单（每轮追问前过一遍）

- [ ] 我这一轮是在哪一层（L1/L2/L3/L4）？
- [ ] 问题数量 ≤ 5 个？
- [ ] 至少有 1 个问题是 trade-off（为什么不选 Y）？
- [ ] 如果用户答含糊词，我是否准备好了追问？
- [ ] **如果用户答"不知道"，我是否准备好了档位 3 的讲解稿？**
- [ ] **这个亮点是否值得我发起一次网络搜索？搜完的信息我怎么融入下一轮？**
- [ ] 我是否在帮用户真正"想透"，而不是在炫技显示我会问？
